# メモリ規律 — 和訳

> 人間用の参考訳。Claude が読み込むのは [SKILL.md](SKILL.md)(英語版)のみ。編集は英語版が先、和訳が追随する。
>
> description 訳: 永続オートメモリ(memory/*.md + MEMORY.md 索引)の書き方と保守の規範。教訓・好み・決定・プロジェクト事実をメモリに保存するとき、既存メモリを更新・整理するとき、想起されたメモリがコードや環境の現状と矛盾しているときに使用。

メモリは将来のセッションが信頼できて初めて役に立つ。ここの各規則は、実際のメモリ監査(2026-07-04)で違反が観測されたものに対応している: リポジトリと重複する仕様書丸写し、索引から漏れたファイル、書いたその日に嘘になったメモリ。

## 入れるもの(入れないもの)

- 1ファイル1事実。耐久的なセットアップ事実と揮発的な状態を混ぜたファイルは部分的に腐る。分割する。揮発的な状態(進捗、検討中の決定)は**書き換えられる前提**の専用ファイルにする。
- リポジトリ・git履歴・CLAUDE.md・チャット履歴が既に記録しているものは保存しない。`docs/` にある仕様・アーキテクチャ・計画は、ポインタ+非自明な差分のみ。コピーはしない。
- 保存するもの: ユーザーの修正指摘と確認済みアプローチ(理由付き)、コードから導出できないプロジェクト事実、外部リソースへのポインタ、そして**担当者付きの未確定事項**(「待ち: 岡田さんがXを確認中」)— 誰が解決するかとセットで記録された未解決点は一級の事実。

## フォーマット

```markdown
---
name: <kebab-case-slug>
description: <1行 — 想起マッチングはこれに対して走る>
metadata:
  type: user | feedback | project | reference
  origin: <model-id>, <YYYY-MM-DD>
---

<事実>

**Why:** <なぜ重要か / どこから来たか>
**How to apply:** <何をどう変えるか>
```

- `origin` にはモデル名と絶対日付を直接書く: セッションIDはログが削除されると追跡不能になるが、frontmatter は消えない。
- 日付はすべて絶対表記。「先週」は3週間後には無意味。
- 関連メモリは `[[name]]` でリンクする。まだ書かれていないメモリへのリンクはエラーではなく「書く価値があるものの印」。
- `feedback` と `project` タイプには Why / How to apply が必須。

## 索引

- メモリファイルを書いた**同じターンで** `MEMORY.md` に1行(`- [Title](file.md) — フック`)を追加する。未索引のメモリは決して想起されない — 存在しないのと同じ。
- 1メモリ1行、内容は書かない。ダッシュの後のフックは「そのファイルを開く理由」として書く。

## 保守 — 同一セッション則

- 直前に観測した事実と矛盾するメモリは、**気づいたそのセッション・そのターンで**更新または削除する — 「あとで直す」印を付けて放置しない。陳腐化したメモリは無いより悪い: 信頼された上で間違っている。
- ほぼ重複の新規作成より既存ファイルの更新。履歴が重要なら日付付き訂正を追記、不要なら書き直す。
- 誤ったメモリ → ファイルと索引行を削除。置き換えられた計画 → 削除するか、置き換え先ドキュメントへのポインタに縮める。
- 保存前に、そのトピックを既にカバーするファイルが索引にないか確認する。
