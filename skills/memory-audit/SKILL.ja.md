# メモリ監査 — 和訳

> 人間用の参考訳。Claude が読み込むのは [SKILL.md](SKILL.md)(英語版)のみ。編集は英語版が先、和訳が追随する。
>
> description 訳: 全永続オートメモリディレクトリ(~/.claude/projects/*/memory/)の一発監査 — 索引漏れ、リポジトリ重複ダンプ、陳腐化・矛盾した事実、形式違反、出所喪失。「/memory-audit」「メモリ監査」「audit memories」等で使用。デフォルトは報告のみ、--fix は機械的修正のみ適用。

全プロジェクトのメモリディレクトリを走査し、各ファイルを [[memory-discipline]] に照らして採点し、重大度順に報告する。`--fix` 付きで呼ばれない限り何も変更しない。`--fix` 付きでも末尾に列挙した機械的修正のみ適用する。

## 手順

1. **列挙。** `ls -d ~/.claude/projects/*/memory` — 各ディレクトリの `MEMORY.md` と他のすべての `*.md` を全文読む。小さいファイルなので全部読む。サンプリングしない。

2. **索引整合。** ファイル一覧と索引を突合: 各メモリファイルに `MEMORY.md` の行がちょうど1つ、各索引行に対応ファイルがあること。未索引=決して想起されない。宙ぶらりん=404を約束する索引。

3. **帰属。** 各ファイルの `origin:` を確認。`originSessionId` しかない場合はモデルの解決を試みる:
   `grep -o '"model":"[^"]*"' ~/.claude/projects/*/<sessionId>.jsonl | sort | uniq -c`
   (プロジェクトディレクトリ名は `-` で始まるので、グロブには `./` 接頭辞か `--` を付けること)。ログ削除済み=出所喪失。フラグを立て、まだ解決可能なうちに記録する。

4. **ファイル別チェック**(重大度順):
   - **矛盾:** 現状に対して安価に検証できる主張(リポジトリの可視性、ファイルの存在、バージョン、解決済みの「確認待ち」)が今は偽になっているもの。
   - **リポジトリ重複:** プロジェクトが既に持つドキュメントの再記述(見分け方: 自分の正本にリンクしている)。ポインタ+差分にすべきもの。
   - **未索引/宙ぶらりん**(手順2より)。
   - **揮発性の混在:** 耐久的なセットアップ事実と現在状況が1ファイルに同居 — 分割点を指摘。
   - **形式:** feedback/project タイプの Why/How 欠落、非 kebab-case の name、`metadata:` 外の `type`、相対日付、`origin` 欠落。

5. **報告。** プロジェクト別にまとめ、1所見1行: ファイル、欠陥、修正方法。冒頭に1段落の総評(最悪の所見+全体の健全度)。モデル別の傾向は帰属が実際に示す場合のみ言及。最後に修正リストを「機械的」と「ユーザーの判断が必要」に分けて提示。

## --fix(機械的修正のみ)

許可: 索引行の追加、宙ぶらりん索引行の削除、手順3で解決できた `origin:` の補記、文脈から絶対日付が確定する相対日付の変換。それ以外 — 陳腐化メモリの削除、ダンプのポインタ化、ファイル分割 — は判断を要する: 列挙するだけで、実行しない。`--fix` でも frontmatter と日付以外のメモリ本文には決して触れない。
