# 0006: /memory-audit コマンドスキル — 報告のみ+機械的 --fix

(English: [0006-memory-audit-skill.md](../en/0006-memory-audit-skill.md))

## Context(背景)

ADR 0005 はメモリ書き込みの規範を成文化したが、検出は手作業のままだった:
その動機となった 2026-07-04 の監査は手動で行われた(ディレクトリ列挙、
索引とファイルの突合、`originSessionId` → モデルのセッションログ解決、
主張と現状の照合)。ユーザーはこれを再実行可能にしたい(同監査の改善提案3)。

## Decision(決定)

新スキル `skills/memory-audit/SKILL.md`(v1.7.0、en + ja 併設): 手動監査を
再現する手順スキル — 列挙、索引突合、帰属解決、重大度順のファイル別チェック
(矛盾 > リポジトリ重複 > 未索引 > 揮発性混在 > 形式)、プロジェクト別報告。
デフォルトは報告のみ。`--fix` はちょうど4つの機械的修正(索引行の追加/削除、
`origin:` 補記、日付の絶対化)だけを適用し、メモリ本文には決して触れない。
削除・ポインタ化・ファイル分割は常にユーザー向けに列挙する。手動監査で
実際に踏んだ2つの運用上の罠(プロジェクトディレクトリ名が `-` 始まりで
グロブがオプション扱いされる件、セッションログは削除されるので出所解決は
可能なうちに行う件)をスキルに埋め込んだ。

## Alternatives rejected(却下した代替案)

- **手順を `memory-discipline` に統合** — トリガーが誤り: あちらはメモリを
  書く最中に発火する。監査は明示的に呼ばれる一発もの。統合すると全メモリ
  書き込みが監査手順で肥大化する。
- **独立スクリプト(`hooks/` 式 Python)** — チェックの半分は判断を要する
  (これは仕様書ダンプか?この主張は矛盾しているか?)。スクリプトは索引と
  frontmatter しかカバーできず、誤った安心感を与える。機械的な部分はモデルが
  実行するインライン grep として残す。
- **デフォルト自動修正** — パターンマッチでのメモリ削除・書き換えはユーザーの
  ヒューマンインザループ哲学に反する。陳腐化に見えるものが意図的な場合もある。

## Consequences(帰結)

- 監査がコマンドになり、`memory-discipline` とペアでループが閉じる
  (書き込み時の規範+オンデマンドの検出)。
- 報告のみがデフォルトのため、対処されるまで所見は再出現する。それはバグ
  ではなく意図された圧力。
- 天井: リポジトリ重複と陳腐化のチェックは、モデルがセッション内で安価に
  検証できる範囲まで。必要になれば schedule スキルで定期実行化も可能。
