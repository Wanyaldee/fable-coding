# 0005: 全プロジェクトメモリ監査から memory-discipline スキルを作る

(English: [0005-memory-discipline-skill.md](../en/0005-memory-discipline-skill.md))

## Context(背景)

2026-07-04、全6プロジェクトのメモリディレクトリ(16ファイル; 書き手は
Opus 4.8、Fable 5、特定不能な旧世代環境)を監査した結果、品質差はモデルでは
なくハーネス世代に追随していた。具体的な失敗: リンクまで張っている `docs/`
ファイルを丸写しした 3.3KB の仕様書ダンプ; 同ファイルの `MEMORY.md` 索引
からの漏れ; 書かれたその日の出来事で嘘になったメモリ(「Private リポジトリ、
スキル1つ」); そして出所の喪失 — セッションログが削除済みで
`originSessionId` しか手がかりのない3ファイルはモデル帰属不能だった。
最良のセット(Opus 4.8、NEDO-KOSEN)には再利用可能な良いパターンがあった:
1ファイル1事実、Why/How、相互リンク、絶対日付、担当者付きの未確定事項。

## Decision(決定)

新スキル `skills/memory-discipline/SKILL.md`(v1.6.0、en + ja 併設)として
監査知見を書き込み/保守の規範に成文化: 耐久/揮発を分離した1ファイル1事実;
リポジトリが記録するものはコピーせずポインタ+差分; frontmatter の
`origin: <model-id>, <YYYY-MM-DD>` フィールド(セッションIDはログ削除で腐る
が frontmatter は腐らない); `MEMORY.md` の同ターン索引(未索引=決して
想起されない); 同一セッション則 — 観測した現状と矛盾するメモリは気づいた
ターンで更新または削除。監査の改善提案 1・2・4・5 をここに収録; 提案3
(`/memory-audit` コマンド)は別途フォローアップ。

## Alternatives rejected(却下した代替案)

- **`fable-coding` への統合** — メモリ書き込みはコーディングタスクではない。
  保存・想起・矛盾発見の瞬間に発火する、別個のトリガースコープ。
- **フォーマット強制フック** — 規則は判断を要する形(何が「記録済み」か、
  いつファイルを分割するか)。バリデータは frontmatter だけ確認して本質を
  見逃す。検出は次ステップの監査コマンドが担う。
- **提案ごとの独立成果物化** — 提案 2/4/5 は単一の規則であり、独立させると
  3つの文が3つのバージョンバンプをまとって歩くことになる。

## Consequences(帰結)

- プラグインを入れたどの環境でも、将来のセッション(と将来の監査)が帰属・
  信頼できる形でメモリが書かれる。この決定の動機となった監査は、
  `/memory-audit` が出来れば機械的に再実行可能になる。
- `origin` フィールドはハーネス標準の frontmatter の拡張。非対応環境でも
  無害(想起は本文と description を読むため)。
- 天井: スキル発火は description マッチングであり、メモリ書き込みの度に
  保証されるわけではない。同一セッション陳腐化則も、モデルが矛盾に気づく
  ことが前提。
